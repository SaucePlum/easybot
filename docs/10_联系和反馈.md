## 技术支持

### 获取帮助的途径

| 途径 | 适用场景 | 响应时间 |
|------|---------|---------|
| [常见问题 Q&A](./09_常见问题Q&A.md) | 快速排查已知问题 | 即时 |
| GitHub Issues | Bug 报告、功能请求 | 1-3 个工作日 |
| Discussions | 使用讨论、经验分享 | 社区互助 |

### 提交 Issue 前请确认

在提交 Issue 之前，请先完成以下检查：

1. ✅ 已查阅[常见问题 Q&A](./09_常见问题Q&A.md)
2. ✅ 已搜索现有 Issues，确认问题未被报告
3. ✅ 准备好复现步骤和最小代码示例
4. ✅ 提供运行环境信息（Python 版本、操作系统、SDK 版本）

### Issue 模板

```markdown
## 问题描述
简要描述遇到的问题

## 复现步骤
1. 执行 `...`
2. 触发 `...`
3. 观察 `...`

## 期望行为
描述你期望的正确行为

## 实际行为
描述实际发生的错误行为

## 环境信息
- Python 版本: `python --version`
- 操作系统: Windows / Linux / macOS
- SDK 版本: pip show easybot
- 运行模式: WebSocket / Webhook / Remote Webhook

## 错误日志/堆栈跟踪
在此粘贴完整的错误输出

## 最小复现代码
在此提供可以独立运行的代码片段
```

## 反馈方式

### 功能建议

我们欢迎任何功能改进建议。提交前请考虑：

- **使用场景**: 这个功能解决什么实际问题？
- **影响范围**: 是否会影响现有 API 的兼容性？
- **替代方案**: 是否有其他方式可以实现类似效果？

### 文档反馈

发现文档错误或需要补充的内容：

1. **文档内容不准确** — 在对应文档页面提出修正
2. **缺少示例代码** — 说明需要的场景，我们会补充
3. **结构优化建议** — 提出你的组织思路

## 贡献指南

### 如何参与贡献

EasyBot SDK 欢迎社区贡献！以下是参与方式：

#### 1. 修复 Bug

```bash
# Fork 项目仓库
git clone https://github.com/your-username/easybot.git
cd easybot

# 创建功能分支
git checkout -b fix/fix-issue-description

# 编写测试（遵循 TDD）
# 在 tests/ 目录下编写失败的测试用例
pytest tests/test_your_fix.py -v  # 确认测试失败

# 实现修复
# ... 编写代码 ...

# 运行全部测试
pytest -v

# 格式化代码
black .
flake8 .

# 提交并推送
git add .
git commit -m "fix: 问题描述"
git push origin fix/fix-issue-description

# 创建 Pull Request
```

#### 2. 新增功能

在开始开发新功能之前：

1. 先创建一个 **Issue** 讨论功能设计
2. 等待维护者确认方案后再动手实现
3. 遵循项目的编码规范（见下方）

#### 3. 改进文档

- 修正错别字、语法错误
- 补充缺失的使用示例
- 翻译文档到其他语言
- 改进文档结构和导航

#### 4. 编写插件

为社区贡献通用插件：

```python
# plugins/example_plugin.py
"""插件名称 — 一句话描述"""

@bot.on_command(command="/your_command")
async def your_handler(msg):
    """命令说明"""
    pass
```

将插件提交到社区插件库或发布为独立的 PyPI 包。

### 编码规范

EasyBot SDK 遵循以下编码标准：

#### Python 版本

- 目标版本: **Python 3.10+**
- 使用类型提示 (Type Hints)
- 使用 async/await 异步编程

#### 代码风格

```bash
# 代码格式化
black .

# Lint 检查
flake8 .

# 类型检查（如适用）
mypy .
```

#### 关键原则

| 原则 | 说明 |
|------|------|
| **KISS** | 保持简单，避免过度设计 |
| **DRY** | 不重复代码，提取公共逻辑 |
| **SOLID** | 单一职责、开闭原则等 |
| **YAGNI** | 不实现当前不需要的功能 |
| **TDD** | 测试驱动开发，先写测试再写实现 |

#### 代码示例规范

```python
from typing import Optional
from easybot import Bot, Model, Scope


async def example_function(
    bot: Bot,
    guild_id: str,
    *,
    limit: int = 100,
) -> list[Model.Member]:
    """
    函数简短描述。
    
    Args:
        bot: Bot 实例
        guild_id: 频道 ID
        limit: 最大返回数量
        
    Returns:
        成员列表
        
    Raises:
        ValueError: 当参数无效时
    """
    if not guild_id:
        raise ValueError("guild_id 不能为空")
    
    members = await bot.api.get_guild_members(
        guild_id=guild_id,
        after="0",
        limit=limit,
    )
    return members
```

### Pull Request 流程

1. **Fork 并克隆**
   ```bash
   git clone https://github.com/your-username/easybot.git
   ```

2. **创建分支**
   ```bash
   git checkout -b feature/your-feature-name
   # 或
   git checkout -b fix/bug-description
   ```

3. **编写代码和测试**
   ```bash
   pytest -v                    # 确保所有测试通过
   black .                      # 格式化代码
   flake8 .                     # 代码风格检查
   ```

4. **提交**
   ```bash
   git add .
   git commit -m "type: 简短描述"
   ```
   
   **Commit Message 格式**:
   - `feat:` 新功能
   - `fix:` 修复 Bug
   - `docs:` 文档变更
   - `style:` 代码格式调整
   - `refactor:` 重构
   - `test:` 测试相关
   - `chore:` 构建/工具相关

5. **推送并创建 PR**
   ```bash
   git push origin feature/your-feature-name
   ```
   然后在 GitHub 上创建 Pull Request。

6. **等待 Code Review**
   - 维护者会审查你的代码
   - 可能要求修改，请及时响应
   - 通过审查后会被合并
